# 软件开发模型
软件生存周期模型又称软件开发模型或者软件过程模型
软件过程式制作用软件产品的一组活动以及结果
### 主要活动
1. 软件描述
2. 软件开发
3. 软件有效性验证
4. 软件进化

面向规格说明的软件开发
```
面向规格说明的软件开发是**以形式化或半形式化的规格说明为核心**的开发方法，先明确系统 “做什么”（功能、性能、约束等）的精确描述，再基于该规格进行设计、实现与验证，核心是 “规格先行、依规格开发”
### 核心核心特征

1. 规格说明为核心：规格说明是开发的基础，需明确、无歧义，可采用形式化语言（如 Z 语言、VDM）或结构化文档（如用例图、状态机）编写。
2. 先定义 “做什么” 再考虑 “怎么做”：优先明确系统需求和行为约束，避免过早陷入具体实现细节。
3. 强调可验证性：规格说明需具备可测试、可证明的特性，方便后续验证系统是否符合需求。
4. 迭代细化规格：从高层抽象规格逐步分解为低层详细规格，确保每个阶段的规格都完整且一致。
```

### 用例视图
描述系统的功能需求，明确 “系统为用户做什么”。
用例、参与者、用例图，聚焦用户与系统的交互场景
- 核心负责人：**产品经理 / 需求分析师**
- 关键参与人员：客户 / 用户代表、业务分析师
### 逻辑视图
描述系统的功能结构和逻辑组件关系，解决 “如何实现需求”。
类、接口、包、协作关系，通过类图、状态图刻画逻辑交互
- 核心负责人：**系统架构师 / 高级设计师**
- 关键参与人员：功能模块设计师、资深开发工程师
### 实现试图
核心作用：描述系统的开发实现架构，聚焦 “如何落地开发”。
核心元素：代码模块、库、工程结构、开发工具，通过组件图、部署图关联代码组织。
- 核心负责人：**技术负责人 / 开发经理**
- 关键参与人员：开发工程师、测试工程师、配置管理员
### 部署视图
描述系统的物理部署架构，解决 “系统运行在什么环境”。
核心元素：硬件设备、网络拓扑、部署节点、组件与设备的映射关系。
- 核心负责人：**运维架构师 / 部署工程师**
- 关键参与人员：系统管理员、网络工程师、硬件采购专员
### 数据视图
核心作用：描述系统的数据存储与流转架构，聚焦 “数据如何存储、访问”。
核心元素：数据模型、数据库表结构、数据流程图、数据访问接口。
- 核心负责人：**数据架构师 / 数据库设计师**
- 关键参与人员：DBA（数据库管理员）、后端开发工程师
# 软件重用  
横向重用：不同的应用领域中的软件元素
纵向重用：在较多公共领域进行重用
# 逆向工程
1. 重构：在同一抽象级别上转换系统描述形式
2. 设计恢复：借助工具从已有程序中抽象出有关数据设计、总体结构设计和过程设计等方面信息。
3. 再工程：修改或重构已有系统
4. 正向工程：从现有系统恢复设计信息
