# 概述
是描述某一个特定应用领域中系统组织方式的惯用模式。
体系结构风格定义一个系统家族，即一个体系结构定义一个词汇表和一组约束。
**词汇表** 包含一些构件和连接件类型
**约束** 指出系统是如何将这些构件和连接件组合起来的
# 数据流风格
所有数据按照流的形式在执行过程中前进，不存在结构的反复和重构。
## 批处理
从数据库、文件等数据媒介读取大量记录，用某种方式处理数据，修改后的形式协会数据。
数据必须是完整的
**应用场景：**
日志分析
计费应用程序
数据仓库
## 管道过滤器
每个构件都有一组输入和输出

# 调用返回风格
## 主程序、子程序风格
一般次啊用单线程控制，把问题划分为若干处理步骤，主程序的正确性取决于它的子程序正确性

## 面向对象风格
数据的标识和他们相应操作被封装起来。
## 层次风格
每一层为上一层提供服务，使用下一层服务
# 独立构件风格
事件的触发者不知道哪些构件会被事件影响， 互相保持独立
## 进程通信风格
进程间消息传递的方式可以是点对点、异步或同步方式。以及远程过程调用
## 事件驱动
当某个事件被触发时，系统自动调用在这个事件中注册的所有哦过程。
断掉调试，公众号订阅  
## 虚拟机风格
当底层不支持上层时，在两者之间加入一层虚拟机做模拟仿真，消除硬件和软件之间的差异
### 解释器风格
包含解释引擎，被解析代码的存储区，记录解释引擎当前工作状态的数据结构，以及一个记录源代码被解析执行进度的数据结构
效率低

## 仓库风格 数据为中心
数据共享风格
### 数据库系统
构件分为中央共性数据源、独立处理单元。
ide集成开发环境。
包含一个数据仓库和若干其他构件
注册表、剪贴板  也是数据库风格
### 黑板系统
多了控制单元，用于监控
一个或多个构件以被动触发的方式以不确定的顺序去更新共享数据存储区
语音识别系统

### 超文本系统
超文本系统中出现的构件以网站链接方式互相连接，用户可以在构件之间进行按照人类联想思维方式任意跳转到相关构件
## 闭环控制
是将过程输出的指定属性维护在一个特定参考值，将事务处理成输入、加工、输出、反馈、再输入一个持续的过程模型。
空调
## C2架构风格
用连接件
1. 系统中的构件和连接件都有一个顶部和底部
2. 构件的顶部应连接到某连接件的底部，构件的底部则应连接到某连接件的顶部。构件和构件之间不允许直接连接
3. 一个连接件和任意数目构件和连接件连接
4. 当两个连接件直接连接，必须由其中一个的底部到另一个顶部

## 二层C/S架构
为单一服务器且以局域网为中心
缺点：
1. 服务器负荷重
2. 数据安全性不好
## 三层C/S架构
增加了一个应用服务器
表示层、功能层和数据层
优点：
1. 逻辑上保持独立性，提供可维护性和可扩展性
2. 更加灵活、有效地选用相应的平台
3. 系统的各层可以并行开发
4. 利用功能层可以有效地隔离表示层与数据层
