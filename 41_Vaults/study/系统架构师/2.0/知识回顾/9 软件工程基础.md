#### 面向规格说明的软件开发
面向规格说明的软件开发是**以形式化或半形式化的规格说明为核心**的开发方法，先明确系统 “做什么”（功能、性能、约束等）的精确描述，再基于该规格进行设计、实现与验证，核心是 “规格先行、依规格开发”
##### 核心核心特征
1. 规格说明为核心：规格说明是开发的基础，需明确、无歧义，可采用形式化语言（如 Z 语言、VDM）或结构化文档（如用例图、状态机）编写。
2. 先定义 “做什么” 再考虑 “怎么做”：优先明确系统需求和行为约束，避免过早陷入具体实现细节。
3. 强调可验证性：规格说明需具备可测试、可证明的特性，方便后续验证系统是否符合需求。
4. 迭代细化规格：从高层抽象规格逐步分解为低层详细规格，确保每个阶段的规格都完整且一致。

#### 模块的耦合性（模块间依赖程度，必考）

耦合性描述模块间的依赖强度，越低越好，考试需掌握 6 种耦合类型的 “定义、示例及优劣”，能判断模块间的耦合等级：

|耦合类型|定义|示例|耦合强度|优劣评价|
|---|---|---|---|---|
|非直接耦合|模块间无直接联系，仅通过主模块间接通信|订单模块与商品模块无直接调用，均由主模块协调|最低|最优，完全符合低耦合原则|
|数据耦合|模块间通过参数传递 “简单数据”（如数值、字符串），无控制信息|订单模块向支付模块传递 “订单金额（100 元）”|较低|良好，推荐使用（仅传递必要数据）|
|标记耦合|模块间通过参数传递 “数据结构”（如数组、结构体），但仅使用部分字段|订单模块向物流模块传递 “订单结构体（含 ID、金额、地址）”，物流模块仅用地址|中等|可接受，避免传递过大的数据结构（仅传递需使用的字段更优）|
|控制耦合|模块间传递 “控制信息”（如标志位、开关量），一个模块控制另一个模块的逻辑|订单模块向库存模块传递 “标志位（1 = 扣减库存，0 = 查询库存）”|较高|不推荐，控制逻辑耦合易导致模块依赖过强（修改控制逻辑需同步修改两个模块）|
|外部耦合|多个模块依赖同一外部环境（如共享全局变量、共享文件）|订单模块与支付模块共享 “全局配置文件”|高|差，外部环境变更会影响所有依赖模块|
|内容耦合|一个模块直接访问另一个模块的内部数据或代码（如修改对方的私有变量）|订单模块直接修改库存模块的 “库存数量” 变量|最高|最差，完全违反封装原则，禁止使用|

#### 模块的内聚性（模块内部功能关联程度，必考）
内聚性描述模块内部功能的关联强度，越高越好，考试需掌握 7 种内聚类型的 “定义、示例及优劣”，能判断模块的内聚等级：

|内聚类型|定义|示例|内聚强度|优劣评价|
|---|---|---|---|---|
|功能内聚|模块仅完成一个独立且明确的功能（最理想）|“用户登录验证” 模块（仅负责验证用户名密码是否正确）|最高|最优，完全符合高内聚原则，推荐设计|
|顺序内聚|模块内的步骤按顺序执行，前一步的输出是后一步的输入|“订单处理” 模块（步骤：验证库存→计算金额→生成订单）|较高|良好，步骤关联紧密，可接受|
|通信内聚|模块内的功能使用相同的输入数据或产生相同的输出数据|“用户信息管理” 模块（功能：查询用户、修改用户、删除用户，均操作用户表）|中等|可接受，数据关联明确，但功能可能不直接相关（如查询与删除）|
|过程内聚|模块内的功能按特定流程执行，但步骤间无数据关联|“系统初始化” 模块（步骤：初始化日志→连接数据库→加载配置，步骤间无数据传递）|较低|较差，步骤仅按流程捆绑，可拆分为独立模块|
|时间内聚|模块内的功能仅因 “时间相近” 而组合（如同时执行）|“系统启动” 模块（功能：启动时同时加载缓存、初始化线程池）|低|差，功能无逻辑关联，应拆分|
|逻辑内聚|模块内的功能逻辑相似但不相关，通过参数选择执行哪项功能|“数据操作” 模块（功能：新增、删除、查询，通过参数选择操作类型）|较低|差，功能混杂，修改一个功能可能影响其他功能|
|偶然内聚|模块内的功能无任何关联，仅因代码片段偶然放在一起|“工具类” 模块（包含日志打印、数据加密、字符串处理等无关功能）|最低|最差，完全不符合内聚原则，必须拆分|

#### 设计原则
| 原则名称        | 核心定义                                      | 实例                                                   |
| ----------- | ----------------------------------------- | ---------------------------------------------------- |
| 单一职责原则（SRP） | 一个类仅负责一个核心功能，避免功能混杂                       | “用户类” 仅负责用户信息管理（属性 + 登录 / 注册方法），不包含订单处理（由 “订单类” 负责）  |
| 开闭原则（OCP）   | 对扩展开放，对修改关闭（新增功能通过扩展类实现，不修改原有代码）          | 支付功能：新增 “银联支付” 时，扩展 “支付抽象类” 实现，不修改已有的 “支付宝支付类”       |
| 里氏替换原则（LSP） | 子类可完全替代父类，且不破坏系统逻辑（子类需兼容父类的方法与属性）         | “VIP 用户类” 继承 “普通用户类” 后，可直接替换 “普通用户类” 参与订单创建，无需修改订单逻辑 |
| 接口隔离原则（ISP） | 客户端不依赖不需要的接口（将庞大接口拆分为细粒度接口，按需实现）          | “支付接口” 拆分为 “在线支付接口” 和 “线下支付接口”，“支付宝支付类” 仅实现 “在线支付接口” |
| 依赖倒置原则（DIP） | 依赖抽象（接口 / 抽象类），不依赖具体实现（高层模块不依赖低层模块，均依赖抽象） | 订单模块不直接依赖 “支付宝支付类”，而是依赖 “支付抽象接口”，可灵活切换支付方式           |
| 迪米特法则（LOD）  | 一个对象应尽量少地了解其他对象（仅与直接关联对象交互，降低耦合）          | “订单类” 仅与 “支付接口” 交互，不直接调用 “支付类” 的内部方法（如日志打印）          |
| 合成复用原则（CRP） | 优先通过 “对象组合” 实现复用，而非 “继承”（避免继承导致的强耦合）      | “订单类” 通过组合 “支付对象” 实现支付功能，而非继承 “支付类”（可灵活替换支付对象）       |
