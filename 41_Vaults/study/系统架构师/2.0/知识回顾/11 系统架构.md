# 软件架构
主要关注软件**构件的结构、属性和交互作用**，并通过多种视图全面描述特定系统的架构
## 架构设计和生命周期
需求分析阶段：收集业务需求，用户需求，非功能需求
设计阶段：软件架构模型描述，软件架构设计与分析方法，设计经验的总结和复用
	体系结构描述语言ADL，支持构建、连接子及其配置的描述语言
	 关注构件间的互联机制
实现阶段：研究基于软件架构的开发过程支持、寻求从软件架构项实现过渡的途径、研究基于软件架构的测试技术
构件组装阶段：支持可复用构件互联

# 基于架构的软件开发方法
## 基于体系的软件设计 ABSD
是体系驱动方法，即构成体系结构的商业，质量和功能需求的组合驱动。
采用视角与视图来描述软件架构，采用用例来描述功能需求，采用质量场景来描述质量需求
是一个自顶向下，递归细化的方法
需求获取来自于：质量目标，系统的商业目标和系统开发人员商业目标
**三个基础：**
1. 功能的分解
2. 通过选择体系结构风格来实现质量和商业需求
3. 软件模板的复用
### 阶段 1：需求分析与架构关注点识别

#### 核心活动
- **需求获取与建模**：
    收集功能性需求（如 “用户登录”“订单创建”）和非功能性需求（如 “响应时间≤100ms”“可用性 99.99%”）；    
    用用例图、用户故事等工具梳理需求，识别对架构设计影响最大的 “架构关注点”（如 “高并发”“数据一致性”）。
- **需求转化为架构约束**：
    将非功能性需求转化为可落地的架构约束（如 “高并发”→约束为 “需设计缓存 + 集群架构”，“数据一致性”→约束为 “需采用分布式事务方案”）。
#### 输出物
- 《需求规格说明书》（含功能性和非功能性需求）；
- 《架构关注点清单》（记录关键非功能需求及对应的架构约束）。
### 阶段 2：架构设计与决策
#### 核心活动
- **架构模式选择**：
    根据架构关注点选择合适的架构模式（如 “高并发电商系统” 选微服务架构，“企业 ERP 系统” 选分层架构）；
    确定核心架构风格（如同步通信 vs 异步通信、集中式 vs 分布式）。
- **组件与交互设计**：
    划分核心组件（如 “用户组件”“订单组件”），定义组件职责；
    设计组件间的交互机制（连接子），如 API 调用、消息队列，并绘制架构图（如 C4 模型、ADL 描述）。
- **架构决策记录（ADR）**：
    记录关键架构决策及理由（如 “选择 Redis 而非 Memcached，因需支持复杂数据结构”）；
    明确架构的约束条件（如 “所有服务必须通过注册中心发现，禁止硬编码 IP”）。
#### 输出物
- 《架构设计文档》（含架构图、组件定义、交互规范）；
- 《架构决策记录（ADR）》；
- 架构原型（可选，用于验证关键技术可行性）。
### 阶段 3：架构验证与评审
#### 核心活动
- **架构原型验证**：
    针对关键架构场景（如 “秒杀高并发”“支付数据一致性”）开发原型，验证架构能否满足需求（如压测原型的 QPS 是否达标）；
    用工具分析架构的潜在问题（如 ADL 工具检查组件循环依赖，性能仿真工具评估响应时间）。
- **架构评审会**：
    组织跨角色评审（架构师、开发、测试、运维），重点验证：
    - 架构是否覆盖所有需求（尤其是非功能需求）；
    - 架构决策是否合理（如技术选型是否匹配团队能力）；
    - 架构是否存在风险（如过度设计、扩展性不足）。
- **架构修正**：
    根据评审意见优化架构（如 “拆分过度耦合的组件”“替换不适合的技术栈”）。
#### 输出物
- 《架构验证报告》（含原型测试结果、工具分析结论）；
- 《架构评审意见与修正方案》；
- 最终版《架构设计文档》。
### 阶段 4：基于架构的开发与实现
#### 核心活动
- **组件分配与团队分工**：
    按架构组件划分开发任务（如 “订单组件” 由订单团队负责，“支付组件” 由支付团队负责）；
    明确团队间的接口对接规范（基于架构设计的组件交互定义）。
- **组件实现与集成**：
    开发团队依据架构规范实现组件（如遵循接口定义、缓存策略）；
    按架构设计的交互机制进行组件集成（如服务注册到指定注册中心，消息发送到指定队列）。
- **持续架构符合性检查**：
    定期审核代码与架构的一致性（如 “是否存在违规跨组件访问数据库”“接口是否与设计一致”）；
    集成测试验证组件交互是否符合架构设计（如 “订单组件调用支付组件是否正常”）。
#### 输出物
- 《开发任务分配表》（组件与团队对应关系）；
- 代码实现（符合架构规范）；
- 《架构符合性检查报告》。
### 阶段 5：架构演化与维护
#### 核心活动
- **架构监控与评估**：
    系统上线后，监控架构关键指标（如组件响应时间、依赖关系变化）；
    评估新业务需求对架构的影响（如 “新增跨境支付” 是否需要扩展现有架构）。
- **架构迭代优化**：
    根据监控数据和新需求，对架构进行增量调整（如 “订单组件拆分出子订单组件”“替换性能不足的连接子”）；
    确保架构演化不破坏核心设计（如 “微服务拆分仍遵循业务域边界”）。
- **架构知识管理**：
    更新架构文档和 ADR，记录演化过程（如 “2024 年 6 月因用户量增长，将缓存架构从单机 Redis 升级为 Redis Cluster”）；
    沉淀架构设计经验（如 “高并发场景的缓存设计模式”）。
#### 输出物
- 《架构监控报告》；
- 《架构演化方案》；
- 迭代后的《架构设计文档》与 ADR。

## 基于体系结构的开发模型ABSDM
### 核心过程
#### 需求分析
收集用户需求、业务需求、质量属性需求
#### 体系结构设计
根据需求和关注点选择合适的体系结构模式（如分层架构、微服务架构、事件驱动架构）；

确定核心风格（如同步通信、异步通信、集中式 / 分布式部署）。

标识构件
1. 生成类图
2. 对类进行分组
3. 把类打包成构件
#### 文档化
产出体系结构规格说明和质量设计说明书
#### 复审
是一个迭代过程
标识出潜在的风险，尽早发现体系结构设计中的缺陷和错误
#### 实现
按体系结构设计拆分开发任务，各团队并行实现组件（如 “订单组件团队”“支付组件团队”）；

遵循体系结构定义的接口规范和交互协议进行组件集成（如服务注册、消息发送）。
#### 演化
系统运行期间，监控体系结构的关键指标（如组件响应时间、依赖关系变化、资源使用率）；

评估新业务需求对现有体系结构的影响（如 “新增跨境支付功能是否需要扩展支付组件”）。
# 软件架构风格
是描述某一个特定应用领域中系统组织方式的惯用模式。
体系结构风格定义一个系统家族，即一个体系结构定义一个词汇表和一组约束。
定义了组件组织方式、交互模式及约束规则
**词汇表** 包含一些构件和连接件类型
**约束** 指出系统是如何将这些构件和连接件组合起来的
## 数据流
所有数据按照流的形式在执行过程中前进，不存在结构的反复和重构。
### 批处理
专注于**大规模数据的离线批量处理**，通过 “数据收集→批量转换→结果输出” 的非实时流程，解决海量数据的汇总、分析、清洗等问题
核心特点是 “高吞吐量、低时延要求、资源集中调度”
应用场景：
- 数据报表生成（如每日 / 每月业务报表、用户活跃度统计）；
- 数据清洗与转换（如离线 ETL，将原始数据标准化后导入数据仓库）；
- 复杂分析（如用户行为画像、商品推荐算法训练）；
- 日志归档与审计（如分析历史日志定位系统漏洞）。
### 管道过滤器
通过 “**过滤器（数据处理组件）+ 管道（数据传输通道）** ” 的组合，实现数据的流式处理
其核心特点是 “**组件高内聚、数据单向流动、流程可组装**”，适用于数据转换流程明确的场景（如日志分析、ETL、视频处理）
应用场景：
- **日志分析系统**：原始日志→解析过滤器→过滤过滤器（保留错误日志）→聚合过滤器（按时间统计）→存储过滤器；
- **ETL 工具**（数据抽取 - 转换 - 加载）：数据库抽取过滤器→清洗过滤器（去重）→转换过滤器（格式对齐）→加载过滤器（写入数据仓库）；
- **视频处理系统**：原始视频→解码过滤器→剪辑过滤器→编码过滤器→输出过滤器（生成 MP4 文件）；
- **Linux 管道命令**：`cat access.log | grep "ERROR" | awk '{print $1}' | sort | uniq -c`（`cat`/`grep`/`awk`/`sort`均为过滤器，`|`为管道）；
- **编译系统**：源代码→词法分析过滤器→语法分析过滤器→语义分析过滤器→代码生成过滤器（生成目标代码）。
## 调用返回风格
基于 “**组件间同步调用与结果返回**” 的交互模式，核心通过 “请求 - 响应” 逻辑组织系统，典型代表为分层架构、客户机 - 服务器架构。
其核心特点是 “同步交互明确依赖、流程可控”，适用于需实时获取结果的场景（如用户操作响应、服务间、同步通信）。
### 主程序-子程序
存在一个**主程序（控制流程）** 和多个**子程序（功能模块）**，主程序按逻辑调用子程序，子程序执行后返回结果给主程序
单机部署，主程序控制整体流程，子程序是功能原子单元；子程序间通常不直接调用
应用场景：
- 本地工具类软件（如文件加密工具、Excel 数据处理工具、计算器）；
- 嵌入式系统程序（如智能设备的控制程序，功能固定、资源有限）；
- 简单的命令行程序（如批量重命名工具、日志统计脚本）；
- 传统单机应用（如早期的桌面版办公软件、本地游戏）
### 面向对象风格
数据的标识和他们相应操作被封装起来。
### 层次风格
每一层为上一层提供服务，使用下一层服务
核心特点是 “**关注点分离、逻辑解耦、易于维护**”，适用于需求稳定、强调规范的中大型企业应用（如 ERP、CRM）
- 中大型企业应用：ERP（企业资源计划）、CRM（客户关系管理）、OA（办公自动化）；
- 传统管理系统：财务系统、人力资源系统、供应链管理系统；
- 接口规范的 B 端应用：需对外提供稳定 API，且业务逻辑迭代不频繁的系统。
## 独立构架风格
一类类以 “**构件独立运行、通过标准化间松耦合交互**” 为核心的架构模式，通过事件、消息等异步步化的通信方式，减少构件间的直接依赖，核心特点是 “**低耦合、高内聚、可替换**”，适用于需求动态变化、需灵活扩展的系统（如插件化应用、分布式服务）
### 事件驱动
当某个事件被触发时，系统自动调用在这个事件中注册的所有过程。
- **构件（事件生产者 / 消费者）**：
    - 生产者：产生事件（如 “订单创建事件”“支付成功事件”）并发布到事件总线；
    - 消费者：订阅感兴趣的事件，接收后执行相应逻辑（如 “支付成功事件” 触发 “订单发货”）。
- **事件总线（Event Bus）**：转发事件的中间件，负责将生产者发布的事件路由到订阅的消费者。
## 进程通信风格
进程间消息传递的方式可以是点对点、异步或同步方式。以及远程过程调用
## 虚拟机风格
“**构建通用执行环境**” 为核心的架构模式，通过定义标准化的语法、语义和执行规则，让 “自定义代码” 在统一框架内运行，核心特点是 “**平台无关、规则可控、灵活扩展**”，典型代表为解释器、JVM（Java 虚拟机）
是指系统包含一个 “**虚拟机核心（执行引擎）** ”，以及一套 “**自定义规则 / 语言**”，用户或开发者可基于这套规则编写 “业务逻辑代码”，由虚拟机核心解释或编译执行，无需关注底层硬件或操作系统细节。
应用场景：
- **跨平台应用开发**：如 Java 开发的企业级应用（可在 Windows/Linux 服务器运行）、Python 脚本（可在不同系统中执行自动化任务）；
- **嵌入式脚本扩展**：如在游戏引擎中嵌入 Lua 脚本（开发者通过 Lua 扩展游戏逻辑，无需修改引擎核心）、在数据库中嵌入 SQL（用户通过 SQL 查询数据，数据库引擎执行）；
- **安全沙箱环境**：如浏览器中的 JavaScript 引擎（限制 JS 代码访问本地文件，防止恶意攻击）、沙箱测试环境（用虚拟机隔离测试代码，避免影响生产系统）；
- **通用编程语言执行**：如 JVM（执行 Java 字节码）、.NET CLR（执行 C# 中间语言）、Python 解释器（执行 Python 字节码）。
### 解释器风格
是指系统针对某一**特定领域**，定义一套简洁的 “规则 / 语法”（如正则表达式语法、配置文件格式），并实现一个 “解释器引擎”，直接逐行或逐条解析并执行符合该规则的 “领域语句”（如正则表达式、配置指令），无需将其编译为中间代码或机器指令。
应用场景：
- **文本处理**：正则表达式（匹配、替换字符串，如日志过滤、手机号校验）、Markdown 解释器（将 Markdown 语法转换为 HTML）；
- **配置解析**：配置文件解释器（解析`application.properties`的`key=value`指令，设置系统参数）、命令行参数解释器（解析`--help`、`-v`等指令）；
- **领域规则验证**：表单验证规则（如用`required:true, max:100`定义输入框规则）、数据格式校验（如 JSON Schema 解释器验证 JSON 结构）；
- **简单脚本交互**：计算器解释器（解析`1+2*3`等数学表达式并计算）、数据库简易查询（如简化版 SQL 解释器，支持基础`SELECT`语句）。
### 基于规则的系统
通过 “**规则库（业务逻辑）+ 推理引擎（执行核心）** ” 的架构，将业务决策逻辑以 “条件 - 动作” 规则形式存储，由推理引擎自动匹配规则并执行，核心特点是 “**逻辑可视化、易维护、可动态调整**”，适用于决策逻辑频繁变化的场景（如风控、审批、故障诊断）

## 仓库风格
数据共享风格
通过集中式数据存储与管理，实现多个独立构件（如业务模块、分析工具）对数据的协同访问与操作，核心特点是 “**数据集中管控、构件松耦合、一致性优先**”，适用于数据为核心资产、需多构件协同处理的场景（如数据库应用、版本控制系统）

仓库风格（Repository Style）是指系统包含一个 “**集中式数据仓库**”（如数据库、文件系统、共享内存），以及多个 “**独立构件**”（如业务处理模块、分析模块），构件不直接交互，而是通过读写仓库中的数据实现协同，数据仓库负责维护数据的一致性、完整性和访问控制。
应用场景：
- **数据库应用系统**：电商订单系统（MySQL 存储订单数据，多构件协同处理）、CRM 客户管理系统（集中存储客户数据，销售 / 售后构件共享访问）；
- **版本控制系统**：Git（仓库存储代码版本，开发者通过提交 / 拉取代码协同开发）、SVN（集中式代码仓库，多团队共享代码）；
- **数据分析平台**：数据仓库（如 Hive）存储海量数据，报表构件、挖掘构件通过查询仓库数据生成分析结果；
- **内容管理系统（CMS）**：集中存储文章、图片等内容，编辑构件、展示构件、审核构件通过仓库协同管理内容生命周期。
### 数据库系统
数据库系统（Database System，DBS）是指由 “**数据库（存储数据）** ”“**数据库管理系统（DBMS，管理数据）** ” 和 “**应用程序（使用数据）** ” 组成的完整系统，用于存储、管理和处理结构化数据，解决 “数据冗余、一致性差、访问效率低” 等问题。
应用场景：
ide集成开发环境。
包含一个数据仓库和若干其他构件
注册表、剪贴板  也是数据库风格
### 黑板系统
是指系统包含一个 “**共享黑板**”（存储问题、中间结果、最终答案）和多个 “**知识源**”（具备特定领域知识的决策构件），知识源通过读取黑板信息、贡献自身知识（更新黑板）的方式，异步协同推进问题求解，无需直接交互。
应用场景：

## C2架构风格
是指系统由**构件**（封装业务逻辑的功能单元）和**连接件**（负责构件间消息路由的通信单元）组成，所有构件按 “逻辑层次” 组织，仅通过连接件传递消息（请求 / 通知），不直接交互，形成 “层次化 + 消息驱动” 的松耦合架构。
1. 系统中的构件和连接件都有一个顶部和底部
2. 构件的顶部应连接到某连接件的底部，构件的底部则应连接到某连接件的顶部。构件和构件之间不允许直接连接
3. 一个连接件和任意数目构件和连接件连接
4. 当两个连接件直接连接，必须由其中一个的底部到另一个顶部
应用场景：
- **企业级应用系统**：ERP 系统（按 “财务层、供应链层、生产层” 划分构件，通过连接件协同）、CRM 系统（“客户层、销售层、服务层” 构件交互）；
- **云服务平台**：SaaS 平台（如钉钉、企业微信，按 “前端层、业务层、数据层” 部署构件，支持多租户动态扩展）、云存储服务（“API 层、业务逻辑层、存储层” 通过连接件通信）；
- 大型在线游戏
- **跨组织协同系统**：政务服务平台（不同部门开发各自构件，通过标准化连接件实现数据互通）、电商生态系统（商家系统、支付系统、物流系统作为独立构件，通过平台连接件交互）。
## 二层C/S架构
二层 C/S 架构（Two-Tier Client/Server）是指系统仅包含 “**客户端（Client）** ” 和 “**服务器（Server）** ” 两个层级，客户端直接向服务器发送请求（如数据查询、业务操作），服务器处理请求后返回结果，客户端负责用户交互与部分业务逻辑（如数据校验、界面展示），服务器专注于数据存储与共享服务（如数据库管理、资源分配）。
为单一服务器且以局域网为中心
缺点：
1. 服务器负荷重
2. 数据安全性不好
## 三层C/S架构
“**客户端（交互层）+ 应用服务器（业务层）+ 数据库服务器（数据层）** ” 的三层结构
表示层、功能层和数据层
优点：
1. 逻辑上保持独立性，提供可维护性和可扩展性
2. 更加灵活、有效地选用相应的平台
3. 系统的各层可以并行开发
4. 利用功能层可以有效地隔离表示层与数据层
## B/S架构
是指系统无需安装专用客户端，用户通过**通用浏览器（如 Chrome、Safari）** 访问**Web 服务器**，Web 服务器处理业务逻辑后与**数据库服务器**交互，最终将结果以网页形式返回给浏览器展示。

# SOA面向服务架构

设计原则：明确定义的接口、自包含和模块化、粗粒度、松耦合、互操作性

主要技术由UDDL、WSDL、SOAP等
UDDL 获取需要的WSDL
- **WSDL**：定义服务 “是什么”（接口、参数、返回值），是服务的 “说明书”；
	- WSDL 是一种**XML 格式的语言**，用于详细描述 Web 服务的接口信息，包括：服务提供的方法（操作）、方法的输入 / 输出参数、数据类型、调用地址、通信协议（如 SOAP）等，是服务提供者与消费者之间的 “接口契约”。
- **SOAP**：定义服务 “如何通信”（消息格式、传输协议），是服务的 “通信语言”；
	- SOAP 是一种**基于 XML 的通信协议**，定义了服务调用的消息格式、传输规则和错误处理方式，用于在分布式系统中实现跨平台的服务交互，可基于 HTTP、TCP 等底层协议传输。
- **UDDI**：定义服务 “在哪里”（注册与发现），是服务的 “黄页目录”。
	- UDDI 是一种**分布式服务注册中心规范**，用于存储服务的元数据（如服务名称、WSDL 地址、提供者信息），支持服务提供者注册服务和消费者查询服务，相当于服务的 “黄页目录”。

## MVP模式
MVP 模式是指将应用程序拆分为 “模型（Model）、视图（View）、表示器（Presenter）” 三个模块，通过 Presenter 作为 View 与 Model 之间的 “桥梁”，实现二者的完全解耦：

- **模型（Model）**：管理数据和业务逻辑（与 MVC 的 Model 一致）；
- **视图（View）**：负责界面展示和用户交互，不直接访问 Model，仅通过 Presenter 传递数据；
- **表示器（Presenter）**：接收 View 的交互事件，调用 Model 处理业务，将结果返回给 View，是核心协调者。
- 本质：对 MVC 的 “解耦强化”—— 在 MVC 基础上消除 View 与 Model 的直接依赖，使 View 更轻量化，Presenter 承担所有业务协调逻辑。
优点：
1. 低耦合
2. 可以更加高效使用模型
3. 复用性好
4. 可测试性好
## MVVM模式
- **视图模型（ViewModel）**：封装 View 所需的数据和交互方法，暴露可观察的数据对象，通过数据绑定与 View 联动，同时调用 Model 处理业务。

## 质量场景
### 1. 刺激源（Stimulus Source）：谁造成的刺激
- **核心定义**：触发质量场景的发起者，即 “谁” 或 “什么系统 / 事件” 引发了对系统的刺激。
- **常见取值**：
    - 外部角色：用户（普通用户、恶意攻击者）、第三方系统（支付接口、物流系统）；
    - 内部组件：系统自身模块（如定时任务、缓存失效机制）、硬件设备（服务器、传感器）；
    - 环境因素：网络（网络抖动、中断）、硬件故障（服务器宕机、硬盘损坏）。
- **关键要求**：明确刺激的发起主体，避免模糊（如 “用户” 需细化为 “10 万并发用户”“恶意攻击者”）。
### 2. 刺激（Stimulus）：响应系统的情况
- **核心定义**：刺激源触发的具体事件或动作，即 “发生了什么” 导致系统需要响应。
- **常见取值**：
    - 正常操作：用户下单、查询数据、提交表单；
    - 异常事件：接口调用超时、服务器宕机、数据校验失败；
    - 峰值负载：秒杀并发请求、大促流量突增、批量数据导入；
    - 攻击行为：SQL 注入、暴力破解、DDoS 攻击。
- **关键要求**：描述具体动作，不笼统（如 “并发请求” 需细化为 “10 万用户同时提交订单请求”）。
### 3. 制品（Artifact）：系统被刺激的部分
- **核心定义**：刺激直接作用的系统组件、模块或数据，即 “系统的哪个部分” 受到影响。
- **常见取值**：
    - 系统组件：应用服务器、数据库、缓存（Redis）、API 网关、消息队列；
    - 业务模块：订单模块、支付模块、库存模块、用户认证模块；
    - 数据资源：用户数据、订单数据、库存数据、配置文件。
- **关键要求**：明确受影响的具体对象，避免范围过大（如 “系统” 需细化为 “订单服务模块”“MySQL 数据库主库”）。
### 4. 环境（Environment）：刺激发生时系统所处的状态
- **核心定义**：刺激发生时，系统的运行状态、硬件条件、网络环境等背景信息，即 “在什么情况下” 发生刺激。
- **常见取值**：
    - 运行状态：正常运行、高并发（如 10 万在线用户）、低负载（如夜间维护）、故障恢复中；
    - 硬件 / 部署：服务器配置（8 核 16G）、集群规模（10 台应用服务器）、部署模式（单机 / 主从 / 集群）；
    - 网络环境：网络延迟（≤10ms）、带宽（1Gbps）、网络稳定性（正常 / 抖动 / 中断）；
    - 数据量：数据库数据量（100 万条订单）、缓存数据量（10 万条商品缓存）。
- **关键要求**：量化环境参数，确保场景可复现（如 “服务器集群” 需明确 “10 台应用服务器 + 2 台数据库主从”）。
### 5. 响应（Response）：刺激产生的结果
- **核心定义**：系统受到刺激后采取的具体动作或行为，即 “系统如何应对” 刺激。
- **常见取值**：
    - 正常处理：接收请求、校验数据、执行业务逻辑、返回结果；
    - 容错处理：故障切换（主库宕机切从库）、请求重试、返回降级结果（如 “暂时无法访问”）；
    - 安全防护：拒绝非法请求、锁定账号、记录攻击日志；
    - 资源调整：自动扩容（增加服务器节点）、限流（拒绝超出阈值的请求）。
- **关键要求**：描述系统的具体行为，不模糊（如 “故障处理” 需细化为 “自动切换至备用服务器，继续处理请求”）。
### 6. 响应度量指标（Response Measure）：量化达标的标准
- **核心定义**：衡量响应是否达标的可量化指标，即 “如何判断系统响应符合要求”。
- **常见取值（按质量属性分类）**：
    - 性能：响应时间（≤500ms）、吞吐量（≥1000 QPS）、并发用户数（≥10 万）；
    - 可靠性：恢复时间（≤1 分钟）、可用性（≥99.9%）、事务成功率（≥99.99%）；
    - 安全性：响应状态码（401/403）、数据加密强度（AES-256）、攻击拦截率（100%）；
    - 可扩展性：扩容时间（≤30 分钟）、功能扩展开发周期（≤3 天）。
- **关键要求**：必须量化（含数值 + 单位），避免 “响应快”“稳定运行” 等模糊描述。
### 可用性
系统正常运行的事件比例，出现故障多久能启用备用系统
提升可用性的策略：
错误检测：心跳，ping

错误避免：服务下线、事务、进程监控

软件架构演化是指**架构随业务需求、技术趋势、环境变化的动态调整过程**，维护是保障架构持续可用的日常管理活动，二者相辅相成，核心特点是 “**渐进式调整、风险可控、持续适配**”，适用于所有长期运行的系统（如互联网产品、企业级应用）
## 分类
按照软件架构的实现方式和实施粒度分类：
基于过程和函数的演化
面向对象的演化
基于组件的演化和基于架构的演化

按照研究方法将软件架构演化方式分类
1. 对演化的支持
2. 版本和工程的管理工具
3. 架构变换的形式方法
4. 架构演化的成本收益分析

软件架构的演化过程是否处于系统运行时期：静态演化、动态演化

## 演化时期

# 可靠性基础知识
可靠性是软件产品在规定条件下和规定时间区间完成规定功能的能力
1. 复杂性
2. 物理退化
3. 唯一性
4. 版本更新快
### 可靠度
规定条件下、规定时间内不发生失效的概念
### 失效强度
单位时间软件系统出现失效的概念
### 平均失效前时间 MTTF 
### 平均恢复前时间 MTTR
从出现故障到修复成功的一段时间
### 平均故障间隔时间 MTBF
包括故障时间到维护设别时间
## 广义可靠性测试和侠义可靠性测试
目的：
1. 发现软件系统的缺陷
2. 为软件的使用和维护提供可靠性依据
3. 确定软件是否达到可靠性定量
