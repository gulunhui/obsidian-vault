数据库是按一定结构组织、存储和管理数据的集合，核心作用是实现 “数据的持久化存储、高效查询与共享”，围绕 “数据模型、数据库管理系统、数据库系统” 三大核心构成

# 数据库体系结构
## 三级模式
**外模式 外部视图** 数据库用户（应用程序）可见的数据视图，是模式的 “子集”，仅包含用户所需的表、字段及关系
**概念模式 概念视图**  数据库的 “整体逻辑结构”，描述所有表、字段、数据类型、表间关系及约束（如主键、外键、唯一性约束），是数据库的核心
**内模式**  数据库的 “物理存储结构”，描述数据在磁盘上的实际存储方式（如文件格式、索引结构、数据压缩方式、存储路径）
## 两级映像
|映像名称|连接的模式|核心作用|对应的独立性类型|
|---|---|---|---|
|**外模式 - 模式映像**|外模式 ↔ 模式|- 定义外模式与模式的对应关系（如外模式的 “用户昵称” 对应模式中 “username” 字段）；<br><br>- 当模式修改时（如新增字段、调整表结构），只需修改该映像（如不将新增字段映射到外模式），无需修改应用程序|**数据逻辑独立性**：应用程序与数据的逻辑结构（模式）解耦，模式修改不影响应用|
|**模式 - 内模式映像**|模式 ↔ 内模式|- 定义模式与内模式的对应关系（如模式中 “订单表” 对应内模式的 “order.ibd” 文件、聚簇索引结构）；<br><br>- 当内模式修改时（如更换存储引擎、调整索引类型），只需修改该映像，无需修改模式和应用程序|**数据物理独立性**：应用程序与数据的物理存储结构（内模式）解耦，物理存储修改不影响应用|

# 关系数据库的规范化
## 函数依赖
设关系模式 R(U)，U 是属性集，X、Y 是 U 的子集。若对于 R 中任意两个元组（行），只要 X 的值相同，Y 的值就一定相同，则称 **Y 函数依赖于 X**，记为 X→Y。

- X 称为 “决定因素”（左部），Y 称为 “依赖因素”（右部）；
- 本质：X 是 “输入”，Y 是 “唯一输出”，不存在 “同一输入对应多个输出” 的情况。

| 分类          | 核心定义                                 | 示例（关系模式 学号姓名课程号课程名分数，主键为 学号课程号）               |
| ----------- | ------------------------------------ | --------------------------------------------- |
| **完全函数依赖**  | Y 完全依赖于 X，不能依赖 X 的任意子集               | 学号课程号分数：分数需同时依赖学号和课程号，缺一不可（单独学号或课程号无法确定分数）    |
| **部分函数依赖**  | Y 部分依赖于 X，即 Y 可依赖 X 的某一个子集（X 是复合属性集） | 学号课程号姓名：姓名仅依赖学号（X 的子集），与课程号无关，属于部分依赖          |
| **传递函数依赖**  | 若 X→Y、Y↛X 且 Y→Z，则 X→Z（Z 传递依赖于 X）     | 关系模式 学号班级班主任 中：学号班级、班级班主任，则 学号班主任（班主任传递依赖于学号） |
| **平凡函数依赖**  | Y 是 X 的子集（Y⊆X），此时 X→Y 必然成立（无实际意义）    | 学号姓名姓名：姓名是 学号姓名 的子集，属于平凡依赖                    |
| **非平凡函数依赖** | Y 不是 X 的子集（Y⊈X），是实际应用中需要关注的依赖        | 学号姓名：姓名不是学号的子集，属于非平凡依赖                        |
| **候选键依赖**   | X 是候选键（可唯一标识元组的最小属性集），Y 是关系中的任意属性    | 候选键 学号姓名、学号年龄：学号作为候选键，可决定所有其他属性               |
### Armstrong 公理系统
| 公理名称    | 核心内容                            | 示例（若 学号姓名，学号班级）             |
| ------- | ------------------------------- | --------------------------- |
| **自反律** | 若 Y⊆X⊆U，则 X→Y（平凡依赖的依据）          | 学号姓名姓名：姓名是 学号姓名 的子集，由自反律推导  |
| **增广律** | 若 X→Y，且 Z⊆U，则 XZ→YZ（左部右部同时增广属性） | 由 学号姓名，增广 课程号，可得 学号课程号姓名课程号 |
| **传递律** | 若 X→Y 且 Y→Z，则 X→Z（传递依赖的依据）      | 由 学号班级 和 班级班主任，可得 学号班主任     |

|推论名称|核心内容|应用场景|
|---|---|---|
|**合并规则**|若 X→Y 且 X→Z，则 X→YZ|由 学号姓名 和 学号班级，合并得 学号姓名班级|
|**分解规则**|若 X→YZ，则 X→Y 且 X→Z（合并规则的逆）|由 学号姓名班级，分解得 学号姓名 和 学号班级|
|**伪传递规则**|若 X→Y 且 WY→Z，则 WX→Z|由 学号班级 和 班级学期班主任，可得 学号学期班主任|
## 规范化
- **1NF**：仅要求属性原子化，与函数依赖无直接关联；
- **2NF**：需消除 “非主属性对主键的部分依赖”（即所有非主属性需完全依赖于主键）；
- **3NF**：需消除 “非主属性对主键的传递依赖”（即非主属性仅能直接依赖于主键）；
- **BCNF**：需消除 “主属性对主键的部分 / 传递依赖”（即所有函数依赖的左部必须是候选键）。

# 控制功能
## 事务管理
事务（Transaction）是数据库中 “一组不可分割的操作单元”，这组操作要么**全部执行成功**，要么**全部执行失败回滚**，不存在 “部分执行” 的情况。

ACID原则

| 特性名称                 | 核心定义                                                | 实现原理                                                                       | 示例（银行转账：A 账户转 100 元到 B 账户）                                        |
| -------------------- | --------------------------------------------------- | -------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| **原子性（Atomicity）**   | 事务是不可分割的最小单位，要么全做，要么全不做，不允许部分执行后中断                  | 基于 “事务日志”（Undo Log）：若事务执行失败，通过 Undo Log 回滚到事务开始前的状态                        | 若 “B 账户收款” 失败，需回滚 “A 账户扣款” 操作，确保 A 账户余额不变                         |
| **一致性（Consistency）** | 事务执行前后，数据库数据需满足预设的业务规则（如 “余额不能为负”“库存不能为负”），数据状态始终合法 | 依赖业务逻辑（如代码判断 “扣款后余额≥0”）和数据库约束（如主键、外键、CHECK 约束）                             | 转账前 A 余额 500、B 余额 300（总 800）；转账后 A 余额 400、B 余额 400（总 800），总金额保持一致 |
| **隔离性（Isolation）**   | 多个事务并发执行时，事务间相互隔离，一个事务的中间状态不会被其他事务可见，避免数据干扰         | 基于 “锁机制”（如行锁、表锁）和 “多版本并发控制（MVCC）”，控制事务对数据的访问权限                             | 事务 1 执行 A 账户扣款时，事务 2 无法同时修改 A 账户余额，需等待事务 1 完成                     |
| **持久性（Durability）**  | 事务执行成功后，数据修改会永久保存到数据库，即使系统崩溃（如断电、宕机），数据也不会丢失        | 基于 “事务日志”（Redo Log）：事务执行时，先将修改记录写入 Redo Log，再刷盘；系统崩溃后，通过 Redo Log 恢复已提交的事务 |                                                                   |

### CAP 定理（一致性、可用性、分区容错性的权衡）

- **核心定义**：分布式系统中，以下三个特性无法同时满足，最多只能满足其中两个：
    - **一致性（Consistency）**：所有节点访问到的数据是最新的（如修改节点 A 的库存后，节点 B 立即读取到最新值）；
    - **可用性（Availability）**：即使部分节点故障，系统仍能正常响应请求（如节点 A 故障，客户端可从节点 B 读取数据）；
    - **分区容错性（Partition Tolerance）**：系统能容忍网络分区（节点间通信中断），仍能正常运行（分布式系统必须满足 P，否则网络故障会导致系统崩溃）。
###  BASE 理论（CAP 定理的落地补充）

BASE 理论是对 “AP 策略” 的延伸，核心是 “放弃强一致性，追求最终一致性”，为分布式系统提供可落地的设计思路：

- **基本可用（Basically Available）**：系统故障时，允许损失部分功能（如降级），但核心功能可用（如电商大促时，关闭 “商品评论” 功能，确保 “下单支付” 可用）；
- **软状态（Soft State）**：系统允许存在 “中间状态”（如数据同步延迟），不影响整体可用性（如主从同步延迟，从节点暂时存旧数据）；
- **最终一致性（Eventually Consistent）**：经过一段时间后，所有节点数据会自动同步为一致状态（如主从同步延迟后，最终从节点会更新为最新数据）。

类型

| --------------- | ------------------------------------ | ------------------------ | ------------------------------------------------------------------------ |
| **1. 增加冗余字段**   | 在表中新增 “其他表的非主属性”，避免关联查询（最常用手段）       | 高频查询需关联少量字段，且关联表数据变更频率低  | 在 “订单表” 中新增 “用户名”（来自用户表）、“商品名称”（来自商品表）字段，查询订单详情时无需关联用户表和商品表              |

### 通过binglog进行数据同步
#### 完整同步流程（以 MySQL 为例）

1. **主节点记录写操作到 binlog**：
    
    当主节点执行写操作（如用户下单），会先将操作逻辑记录到`binlog`（顺序写入，性能高），再更新本地数据。
    
2. **从节点 IO 线程拉取 binlog 并写入 relaylog**：
    
    从节点的`IO线程`通过网络连接主节点，请求同步`binlog`；主节点通过 “dump 线程” 将`binlog`内容发送给从节点；从节点`IO线程`将接收的内容写入本地`relaylog`。
    
3. **从节点 SQL 线程重放 relaylog 同步数据**：
    
    从节点的`SQL线程`读取`relaylog`中的操作记录，按顺序执行（如同样执行 “UPDATE user ...”），确保从节点数据与主节点一致。
    
基于sql语句的复制：如果由变量，可能会造成不一致

| 复制模式      | 核心逻辑                                                    | 数据一致性              | 写入性能                    | 可用性                  | 适用场景                    |
| --------- | ------------------------------------------------------- | ------------------ | ----------------------- | -------------------- | ----------------------- |
| **异步复制**  | 主节点执行写操作后，立即返回 “成功”，无需等待从节点同步（`binlog`异步发送）             | 最终一致性（从节点数据滞后，毫秒级） | 最高（无等待开销）               | 高（从节点故障不影响主节点）       | 互联网非核心业务（如商品评论、用户日志）    |
| **半同步复制** | 主节点执行写操作后，需等待 “至少 1 个从节点将`binlog`写入`relaylog`”，才返回 “成功” | 弱一致性（至少 2 节点数据一致）  | 中（等待 1 个从节点确认，增加几十毫秒延迟） | 中（若所有从节点故障，主节点写操作阻塞） | 电商订单、用户信息（需一定一致性，又需高可用） |
| **全同步复制** | 主节点执行写操作后，需等待 “所有从节点执行完`relaylog`操作并返回确认”，才返回 “成功”      | 强一致性（所有节点数据实时一致）   | 最低（等待所有从节点，延迟高）         | 低（任一从节点故障，主节点写操作阻塞）  | 金融核心交易（如转账、余额更新，需强一致）   |

#### 读写数据步骤

| 案名称                         | 核心逻辑                                                                    | 适用场景                    | Redis 实现方式                                            |
| --------------------------- | ----------------------------------------------------------------------- | ----------------------- | ----------------------------------------------------- |
| **更新数据库后删除缓存（Cache Aside）** | 1. 更新 MySQL 数据；2. 删除 Redis 对应 Key（下次查询时从 MySQL 加载并更新 Redis）             | 写少读多，缓存数据需多表关联计算（如商品详情） | 用 Redis DEL 命令删除 Key，简单高效                             |
| **延迟双删**                    | 1. 先删除 Redis Key；2. 更新 MySQL 数据；3. 延迟 100ms 再次删除 Redis Key（避免并发更新时旧值写入） | 高并发读写场景（如库存更新）          | 结合 Redis DEL 和定时任务（如 Java 的 ScheduledExecutorService） |
| **Redis 事务 + Lua 脚本**       | 将 “更新数据库 + 更新缓存” 封装为原子操作，用 Lua 脚本确保 Redis 操作的原子性                        | 一致性要求高的场景（如用户积分）        | 用 Redis EVAL 命令执行 Lua 脚本，避免并发导致的中间状态                  |

#### 过期策略
1. 惰性删除
	查询key才检测是否过期
2. 定期删除
	每隔一段时间对数据库做一次检查，删除里面的过期key
3. 内存淘汰机制
	当内存不足时，Redis 会根据配置的淘汰策略删除 Key，常用策略包括：
- `volatile-lru`：从设置了过期时间的 Key 中，删除最近最少使用的 Key；
- `allkeys-lru`：从所有 Key 中，删除最近最少使用的 Key（适用于混合存储场景）；
- `volatile-ttl`：从设置了过期时间的 Key 中，删除剩余过期时间最短的 Key；
- `noeviction`：默认策略，不删除任何 Key，内存不足时拒绝新写入请求（可能导致业务故障

#### 缓存异常
##### 缓存穿透
key不在redis中。
可能原因：
1. 恶意攻击
	1. 针对ip限制访问次数
	2. 通过应用程序检查key
	3. 使用布隆过滤器
2. 大量数据库有但是redis没有key
	1. 预热redis
	2. 再前端控制流量
	3. 将value设置为null
解决：缓存空值、布隆过滤器、请求校验
##### 缓存击穿
热点key设置了太短的过期时间
1. 设置较长的过期时间
2. 使用分布式锁
解决：热点key永不过期、互斥锁、二级缓存
##### 缓存雪崩
大量key同时失效
可能原因：
1. redis故障
	1. 主从复制
	2. 服务降级，服务熔断
2. 采用了相同的过期时间
	1. 过期时间加上随机值
解决：过期时间随机化、缓存集群高可用、熔断降级、数据预热
