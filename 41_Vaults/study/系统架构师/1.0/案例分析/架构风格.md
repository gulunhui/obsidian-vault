# 经典架构风格
## 数据流风格
### 批处理
特点：
* 每个处理步骤是一个独立的程序
* 数据必须是完整的，以整体方式传递
* 处理步骤之间没有并行性，每个步骤必须依次执行
* 不适合用来设计交互式应用系统
* 处理流程固定，调整顺序需要重启
### 管道过滤器
特点：
* 不适合用来设计交互式应用系统
* 过滤器可以并发调用，性能好
* 由于没有通用的数据传输标准，因此每个过滤器都需要解析输入数据和合成数据，批处理风格也有这样的特性
* 适合流式数据处理
* 处理流程固定，调整顺序需要重启
## 调用、返回风格
### 面向对象风格
特点：
* 基于数据抽象和面向对象的思想
* 数据的表示和相关操作被封装在对象中
* 如果一个对象要调用另一个对象，则必须要知道它的标识和名称
### 层次结构分割
特点：
* 系统按照功能分为若干层，每一层为上层提供服务，并作为下层服务
* 内部层接口只对相邻层可见
* 很难找到一个合适和正确的层次划分方法
* 传输数据需要经过多个层次
* 多层难以调试
### 客户端/服务器
两层C/S 胖客户机（安全性较差）
	适应性高灵活性高
	功能构件隔离，集中任务处理，提高开发效率
	————————
	开发成本高，客户端复杂
	界面风格不统一
	软件移植和升级困难
	系统安全难以保障
三层C/S 瘦客户机
	合理划分功能
	安全性更高，通过功能层隔离
	支持并行开发，各层可选择适合的开发语言和平台
	—————————
	层间通信效率影响整体性能
	设计时需慎重考虑通信方式、频度和数据量
B/S 架构
	安装升级维护集中在服务器，降低客户端维护成本
	用户仅需浏览器即可操作，使用简单
	开发成本低，易于快速部署和更新
	——————————
	对动态页面和复杂数据库功能支持较弱
	数据查询速度低，动态交互性不足
	安全性难以控制，面向公开网络易受攻击
## 独立构件
### 事件系统体系风格
特点：
* 隐式调用
* 构件不直接调用一个过程，而是触发或广播一个或多个事件
* 事件发布者不需要知道哪些构件响应事件
* 构件放弃了对计算的控制，完全由系统来决定

## 虚拟机风格
### 解释器
特点：
 * 自定义、灵活性好
* 执行效率低
### 基于规则的系统
特点：
 * 自定义、灵活性好
* 执行效率低
## 仓库风格
### 数据库系统
特点：
* 便于客户共享大量数据，而不必关心数据是如何产生的、由谁提供的及通过何种途径来提供
* 各知识源通过中央数据单元交互、数据与处理解耦
* 支持交互式处理
### 黑板系统
特点：
* 分布式问题求解
* 共享知识库存储问题的当前状态和已经得到的解决方法
* 数据与处理解耦，知识源是相对独立的，他们负责解决问题的不同方面
* 利于处理复杂、非结构化的问题，如语音识别
# 面向服务架构 SOA
SOA 对于实现企业资源共享，打破信息孤岛
## 主要协议和规范
### UDDI 统一描述，发现和集成
提供一种服务发布，查找和定位的方法
### WSDL
web服务描述语言是对服务进行描述的语言
### SOAP
简单对象访问协议定义了服务请求者和服务提供者之间的消息传输规范
### REST
表述性状态转移，REST
1. 资源，以资源为中心
2. 表述，资源的状态
3. 状态转移，应用状态保存在客户端，资源状态保存在
4. 超链接
## 设计原则
* 无状态
* 单一实例
* 明确定义的接口
* 自包含和模块化
* 粗粒度
* 服务间的松耦合
* 重用能力
* 互操作性、兼容和策略声明
# 微服务架构
是SOA的一种实现方式
微服务的优势：
1. 通过分解巨大单体式应用为多个服务方法解决了复杂性问题。
2. 让每个服务能够独立开发，开发者能够自由选择可行的技术。
3. 微服务架构模式是每个微服务独立的部署
4. 微服务使得每个服务独立扩展

挑战：
1. 并发所有服务都能转为微服务
2. 部署较单体架构更加复杂
3. 性能问题：由于微服务注重独立性，互相通信时只能通过标准接口，可能产生延迟或调用出错
4. 数据一致性问题：作为分布式部署的微服务，在保持数据一致性方面比传统架构更加困难
