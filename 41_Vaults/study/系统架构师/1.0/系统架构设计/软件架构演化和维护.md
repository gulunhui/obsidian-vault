# 分类方法
按照实现方式和实施粒度分类：基于过程和函数的演化，面向对象的演化，基于组件的演化和基于架构的演化
按照研究方法分类：第一类是对演化的支持，第二类是版本和工程的管理工具，第三类架构变换的形式方法，包括系统结构和行为变化的模型，以及架构演化的重现风格，第四类架构演化的成本收益分析
系统是否处于运行时期：动态演化和静态演化
## 演化时期
1. 设计时演化：发生在体系结构模型和代码编译之前
2. 运行前演化：执行前，编译之后
3. 有限制运行时演化：满足要求后才能演化，系统较安全
4. 运行时演化：最难实现
## 静态演化
涉及到设计时演化，运行前演化
正交软件体系架构演化过程：
1. 需求变动归类，判断组件重用情况
2. 制定架构演化计划
3. 修改、新增或删除组件
4. 更新组件之间的互相作用
5. 产生演化后的软件架构，作为系统更新的详细设计方案和实现基础
## 动态演化
软件内部执行所导致的体系结构变化
软件系统外部的请求对软件进行的重配置
### 类型

### 动态演化的内容

# 软件架构演化原则
演化成本控制原则（ECC）
	演化成本要小于开发成本‘
进度可控原则(SC)
	要在预期时间内完成
风险可控原则(RC)  
	经济风险、时间风险、人力风险、技术风险、环境风险要在可控范围内
主体维持原则（AIG）
	对称稳定增长原则，所有其他因素必须与软件演化协调，开发人员，销售人员、用户必须熟悉软件演化内容，从而令人满意的演化。
系统总体结构优化原则（OWS）
	让演化后的系统整体结构更加合理
	检查系统的整体可靠性和性能指标
平滑演化原则
	演化速率趋于平稳，更新率相对固定
目标一致性原则 OC
	阶段目标和最终目标要一致
模块独立演化 LC
	各模块自身演化最好互相独立
影响可控原则 IL
	如果一个模块发生变更，给其他模块带来的影响要在可控范围内
复杂性可控原则
	演化必须要控制架构的复杂性，从而进一步保证软件的复杂性在可控范围内
有利于重构原则
	演化后的软件更便于重构
有利于重用原则
	维持甚至提供架构的可重用性（构件化）
设计原则遵从性原则
	不与架构设计原则冲突
适应新技术原则
	要独立于特定的技术手段
环境适应性原则
	演化后的软件版本能够适应新的硬件环境于软件环境
标准依从性原则
	演化不会违背相关质量标准（国标，行标）
质量向好原则 QI
	关注的某个质量指标或某些质量指标的综合效果变得更好或者更满意
适应新需求原则
	架构演化要容易适应新需求，不能降低原有架构适应新需求的能力

# 架构演化评估方法
评估过程：演化的过程已知评估、演化过程未知的评估
## 演化评估目的
演化过程已知的评估其目的在于通过对架构演化过程进行度量，比较架构内部结构上的差异以及由此导致的外部质量属性上的变化，对演化过程中相关质量属性进行评估。
Q的组合：圈复杂度、扇入扇出度，模块间的耦合度，模块的响应，紧内聚度，松内聚
可维护性距离计算方法：Dm(A,B) 值越大，可维护质量属性差距越大
## 演化过程未知的评估
通过演化前和演化后的架构，架构质量属性变化
