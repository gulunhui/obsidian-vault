# 反规范化设计
优点：避免进行表之间的连接操作，可以提高数据操作性能
缺点：数据重复存储，浪费了磁盘空间。会产生数据的不一致问题

增加冗余列：查询时避免连接操作
增加派生列：增加的列可以通过其他列生成。避免查询时的计算量，从而加快查询
重新组表：重新组表指如果用户需要查看两个表连接出来的结果，则把这两个表重新组成一个表
水平分割表：按记录进行分割，把数据放到多个独立表中
垂直分割表：对表进行分割，将主键和一部分列放到表中。

# 分布式数据库优化
## 主从复制
好处：
1. 作为数据的热备
2. 架构的扩展
3. 读写分离
### 主从复制的方式：
binlog进行数据同步
1. 主库创建binlog日志
2. 从库连接主库，获取binlog日志
3. 发送binlog中的日志到从库
4. 将日志存放到relay log
5. 读取relay log中的日志信息
6. 将数据库更新事件在存储引擎中执行
### binlog日志的模式：
1. 基于sql语句的复制
	优点：日志量比较小
	缺点：如果绑定了本地变量，那么可能会不一致
2. 基于行的复制
	优点：主从数据复制完全一致
	缺点：日志量比较大
3. 混合复制
### mysql主从复制模式
全同步复制
	等待所有从库都执行了该事务才会返回给客户端。延迟比较大
半同步复制
	主库只需要等待一个节点从binlog中收到该请求，并保存到relay log即可。优点：节约时间
异步同步复制
	主库执行完客户端提交的事务后立即将结果同步给客户端。

## 分表
也叫分片，解决并发能力、IO性能提升。将一张大表分成若干小表，业务同时访问表。
垂直切分：将一个大表切分为多个表。
水平切分：每个表都有相同列。
## 分库
原本将一个实例上的数据分开存放到不同的实例上。
要解决使用的透明度。不能让应用知道分库分表的复杂度太高，造成耦合太高
数据库代理：延迟比较大
jar包代理：延迟比较小，升级比较困难

# 分布式缓存技术
## redis
是基于内存的数据库，在执行任何操作时，一般没有磁盘的低速操作。
以key-value的形式保存
### 数据类型
string：基本类型。可用于缓存层或计数器，如视频播放
hash：代理string节省空间
set：无序集合，每个值不能重复
list：双向链表结构，可以模拟栈和队列
zset：有序集合，每个元素有个分数
pub/sub：可以用做消息队列
### 读写步骤
读数据：
1. 根据key读缓存
2. 读取成功则返回
3. 若key不在缓存中，根据key读数据库
4. 读取成功，写入缓存
5. 成功返回
写数据：
1. 根据key写数据库
2. 更新缓存
### 过期策略
1. 惰性删除
	查询key的时候才对key进行检测
2. 定期删除
	每隔一段时间做一次检查，对所有过期的key删除。
3. 内存淘汰机制
	最近最少使用
	最不经常使用
	随机淘汰
	生存时间淘汰
### redis持久化
如果服务器待机，内存数据库中数据将全部丢失。需要对数据进行持久化
#### 策略
RDB内存快照：把当前内存中的数据集快照写入磁盘。恢复时将快照文件直接读取到内存里
AOF日志：通过持续不断的保存redis服务器所执行的更新命名来记录数据库状态。类似于mysql和binlog。恢复数据时需要从头开始回放更新命名
### 缓存异常
1. 缓存穿透
	大量访问到不存在的key
	可能原因：
	1. 恶意攻击
	解决方案：
	* 针对来源是比较少的请求来源ip，主动限制访问次数
	* 应用程序检查key的合法性。
	* 使用布隆过滤器
	2. key在数据库中，redis没有。
	解决方案：
	* 预热redis
	* 在最前端进行流量控制
	* 如果数据库里也没有的key，也要在redis中设置key，使其为null
	#### 布隆过滤器
	不存在则一定不存在，如果存在，则不一定真的存在
2. 缓存雪崩
	大量的key同时失效。
	可能原因：
	1. redis故障：使用主从复制，服务降级，熔断
	2. 采用了相同的过期时间：过期时间增加一个随机值
3. 缓存击穿
	热点key设置了太短的过期时间
	解决方案：
	* 设置较长过期时间。
	* 使用分布式锁
采用集群方式
服务降级、服务熔断、请求限流
